Overview

In this guide we are going to be looking at how we look at other people's work to understand, study, and use a piece of technology as we work to develop our own.

Specifically, we are going to be looking at WYSIWYG editors get on the web see if we can discover what technology is primary used to enable rich text editing.

Understanding our Requirements

The example Project

For this tutorial we are going to be looking at building a website that involves a text field. We want users to be able to bold and italicize text, and add spacing which will be recorded in the submission.

Identifying key users

Poets are in dire need of inspiration. A simple Haiku won't do.

It's very important to talk to all your stakeholders. In this example we need to identify potential end users (Potential Poet Inspirers, or PPIs), as well as potential end recipients (Potential Poet Recipients, or PPRs).

Quick Tip: Using Highly Obscure Acronyms (or HOAs) in your projects will cause your work to seem much more important and cool.

If you don't understand everyone who uses your application, service, or product, it won't do well. Groups of people are very different, and people will be scared away from a product that doesn't clearly understand their needs or wants. So before you embark upon a project, make sure you talk to and have people ready that you can meet with and receive feedback at every step of the project.

For this reason, it's often a really good idea to build a product that serves your own needs. This is part of why software developers have an embarrassment of riches in regards to high quality tools and technologies but many other people such as business leaders, researchers, artists, call centers etc. are often using old or frustrating software.

Playing around with our example application

Currently what I have set up for this project in a <textarea> . This serves the purpose of accepting text input and return the data when the user clicks submit. This is to goto solution for text input around the web.

Some examples:

reddit

Bitbucket

There's many examples out there of <textarea> being used. Why is it used so often? There are some key reasons:

  1. Highly marketed-- almost all web developers know about it. It's the goto when it comes to multi line input
  2. Highly compatible, works everywhere, mobile phones, ipads, etc.
  3. Integrates well with <forms> -- it returns the text input

Play around with textarea and try to break it. The goal of this tutorial is to identify what makes a product or solution good so studying textarea is one place you can identify some of these things. Open up any one of these sites textarea input and play around with it.

Notice:

  1. Spelling is automatically checked on most browsers in textarea input
  2. Undo hotkeys are available, try pressing control+z or cmd+z and you can undo your actions
  3. Try clicking a link after entering input and press the back button. The input might still be there. Cool!
  4. Try repeating your experience on other platforms. Is it pretty consistant/good?
  5. Try pasting stuff in, the formatting is automatically cleared and text is extracted. Try selecting text, copy, paste. Selecting and dragging words work. Trying copying and pasting the whole surrounding page.
  6. You can resize the text area as needed and it scrolls

So textarea is pretty solid, but one key drawback: It's unformatted.

One solution that many developer-centric sites use into to allow the use of a markup language such as markdown.

Advantages of this solution:

  1. Get to use <textarea> which is rock solid
  2. Uses the WYSIWYM -- What you see is what you mean paradigm -- which some people (developers) sometimes like more
  3. The formatting isn't as accessible -- could be good or bad depending on your use case

Disadvantages of this solution:

  1. Users might accidentally format their input when they don't mean to by using special markup characters
  2. Formatting isn't as accessible to users
  3. Can't copy/paste formatting (unless the formatting is plain markup text)

Is this a good fit for our use case?

Formatting is an essential part of our application according to our PPRs, and our PPIs might not be too keen to learn and understand a new markup language to use our application. A markup language might also not be as good as reflecting what the end message will look like (since it's using the WYSIWYM and not the WYSIWYG paradigm).

For this project a WYSIWYG/rich text editor might be a better fit.

Conclusion: using this knowledge in our solution

Most sites and libraries appear to use contenteditable, which is an html5 technology in which the web browser handles much of the interaction required for rich text editors.

By looking at other sites, we can see how it's used and some of its limitations. With this we can make a decision about its use in our application. We might decide to allow the bold and italics and use javascript to strip out all other formatting like tumblr, or keep all formatting like quora. Many sites use buttons to allow users to select their text and perform common actions which is something to consider.

In summary, by examining how other sites treat certain behaviors we can get a better sense of how a technology works and how it behaves. By looking at libraries we can see how software can be used to overcome shortcomings that the technology lacks.
